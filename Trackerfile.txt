[PROJECT SPEC — decisions carried over from the original build + this rebuild]

## Identity (unchanged from v1)
archetype: Rebel/Outlaw
fonts: Bebas Neue (heading) + Inter (body)
base palette: ink #0B0B0E, panel #17171C, accent #00D9FF (cyan), text #F2F2F2, muted #8A8A92

## NEW: Dynamic full-site color theming
Selecting a colorway must re-tint the ENTIRE site (not just the bike image):
  - Metallic Black  → base palette above (default state)
  - Ice Storm Cyan   → accent intensifies, background glow/particles shift cyan
  - Vivid Violet     → accent becomes #8B5CF6, background glow/particles shift violet
Implementation approach: CSS custom properties on :root, swapped via a small
React context (`ThemeProvider`), animated with GSAP color tweens (not an instant
snap) so the transition itself feels cinematic. The R3F particle/glow layer reads
the same context so WebGL color updates in sync with the DOM.

## Reference sites (aesthetic target)
- https://wonderland.studio/ — dark "spacey" backdrop, white line/particle fields,
  speed-up-on-click micro-interaction, GSAP + Three.js
- https://activetheory.net/ — full WebGL environments, GSAP + custom GLSL shaders,
  "build the animation system before the content" philosophy
Neither site is being copied directly — matching the FEELING (dark, particle-driven,
scroll-scrubbed, shader-based ambient motion) applied to an MT-15 product showcase,
not their literal layouts or content.

## Locked stack (supersedes v1's plain-HTML approach)
- Next.js + TypeScript + React (needed for React Three Fiber)
- Tailwind CSS v4 (carried over, CSS-first @theme config)
- GSAP + ScrollTrigger (core animation engine, scroll-scrubbed reveals + morphing)
- Lenis (smooth-scroll wrapper) — verify early it doesn't break `position: sticky`
  before building anything that depends on sticky (known Locomotive Scroll issue;
  testing whether Lenis shares it)
- React Three Fiber + @react-three/drei (ambient WebGL layer only — particles,
  glow, shader backgrounds, image-distortion transitions — NOT a full 3D bike model)
- Explicitly SKIPPED: Framer Motion (redundant with GSAP, adds bundle weight),
  Scrollama (ScrollTrigger already covers this), D3.js (no data-viz on a spec sheet)

## Imagery approach (supersedes "no real photos" from v1)
No official Yamaha photography (copyright/trademark risk — this is what killed
real-photo use in v1). Instead: AI-generated photoreal images, created by the USER
via ChatGPT/DALL-E using prompts Claude writes. Prompts must be tuned to:
  - The site's color theory (matches whichever colorway section they're for)
  - Accurate MT-15 design cues (naked streetfighter stance, DeltaBox-style frame,
    aggressive LED headlight, muscular fuel tank) WITHOUT referencing or describing
    any specific copyrighted photograph — describe the real vehicle's public design
    characteristics, the way a stock-photo brief would, not a copied image.
Status: prompts not yet written — this is Phase 5 in the plan below.

## ⚠️ CARRIED-OVER PITFALLS (from prior session's analysis — still apply)
💀 OUTSTANDING → Trigger: stacking GSAP + Three.js + heavy image sequences without
lazy-loading → Prevention: load WebGL/3D assets AFTER initial paint; keep first
paint lightweight → Recovery: if Total Blocking Time balloons, defer non-critical
JS with `next/dynamic({ ssr:false })`, or gate heavy scene init behind
IntersectionObserver so it only initializes once scrolled into view. (A reference
portfolio site hit 21s TBT front-loading WebGL — treat this as a hard failure mode,
not a minor perf nit.)
⚠️ MEDIUM → Trigger: ignoring prefers-reduced-motion → Prevention: gate ALL GSAP
timelines AND the R3F canvas itself behind a reduced-motion check — for
reduced-motion users, skip mounting the WebGL canvas entirely (not just pausing
its animations) and fall back to a static styled layout. → Recovery: none needed
if built in from the start; retrofitting is much harder once the canvas is wired
into scroll state.

## Phase plan (high-level — each phase becomes multiple Trackerfile entries as worked)
1. Tag v1 as a recovery point, branch off for the rebuild — DO NOT touch `main`
   until a phase is fully working
2. Scaffold Next.js + TypeScript + Tailwind v4 in the new branch
3. Port design tokens (colors/fonts) into Tailwind theme + CSS custom properties;
   build the ThemeProvider/context for dynamic colorway theming
4. Core layout + copy ported from v1 (hero, philosophy, specs, features, CTA,
   footer) as plain React components first, unstyled-for-motion — motion comes
   after structure exists, not before
5. Imagery: write AI-image-generation prompts (color-matched per section), user
   generates them externally, uploads back, Claude integrates + optimizes
   (next/image, proper sizing, lazy loading)
6. Motion layer: GSAP + ScrollTrigger scroll-scrubbed reveals, Lenis smooth scroll
   (test against sticky positioning immediately), morphing element transitions
7. WebGL ambient layer: R3F + drei particle/glow scene, gated behind
   IntersectionObserver + reduced-motion check per the pitfalls above
8. Dynamic color theming wired end-to-end (DOM + WebGL layer in sync)
9. Performance pass: Lighthouse/TBT check, lazy-load audit, mobile test
10. Deploy: Next.js static export → GitHub Pages (or Actions-based deploy if
    static export can't support the WebGL canvas — verify in Phase 9)
